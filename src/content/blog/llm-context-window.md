当人们谈论大语言模型的能力时，最常被提到的指标是**参数量**。GPT-4 有多少参数，Claude 有多少参数——这些数字代表模型的"容量"，听起来很重要。

但实际使用中，**上下文窗口的大小**往往比参数量更直接影响你的使用体验。

## 上下文窗口是什么？

简单来说，上下文窗口是模型在生成回复时**能同时"看到"的文本长度**。

你发给模型的消息、模型回复的内容、上传的文档——所有这些都会占用上下文窗口。超出窗口大小的内容，模型就"看不见"了，就像你往窗户外面扔了一张纸，它飘走了，你再也捡不回来。

上下文窗口通常用 **token** 来度量。Token 不完全等于字符——英文大约 1 token = 0.75 个单词，中文大约 1 token = 0.5 个汉字。

目前主流模型的上下文窗口：
- GPT-4o：128k tokens（约 10 万汉字）
- Claude 3.5 Sonnet：200k tokens（约 15 万汉字）
- Gemini 1.5 Pro：1M tokens（约 75 万汉字）

## 更长不一定更好：中间迷失问题

有个有趣的研究叫做 **"Lost in the Middle"**（中间迷失）。

研究发现，当上下文很长时，模型对**开头和结尾**的内容注意力更强，而对中间部分的信息处理能力会显著下降。

这意味着：如果你把最重要的指令塞在一份 10 万字文档的中间，模型很可能忽略它。

**实践建议**：
- 重要指令放在最开头或最结尾
- 长文档分段处理，而不是一次性全部输入
- 关键信息重复一次（在开头提一次，结尾再强调）

## Token 的经济学

上下文窗口不只是技术限制，它也直接影响成本。

大多数 API 按 token 计费，包括输入和输出。一个 200k 上下文的对话，比一个 10k 上下文的对话贵 20 倍——即使你问的是同一个问题。

在构建 AI 应用时，**上下文管理**是一个核心工程问题：
- 如何在有限的窗口里放最有价值的信息？
- 哪些历史对话可以压缩或丢弃？
- 什么时候用 RAG（检索增强生成）代替塞入原文？

## RAG 的意义

RAG（Retrieval-Augmented Generation）本质上是解决上下文限制的工程方案。

与其把整本书塞进上下文，不如先用向量数据库找出最相关的段落，只把这些段落放进上下文。这样既节省 token，也减少了"中间迷失"问题。

未来随着上下文窗口持续扩大，RAG 的必要性会降低，但它背后的**信息检索**思维不会过时——如何找到最相关的信息，永远是关键问题。

## 对普通用户的启示

理解上下文窗口，能让你更聪明地使用 AI 工具：

1. **新话题开新对话**：旧对话积累的无关上下文会干扰新问题的回答质量
2. **粘贴长文档时，先告诉模型重点在哪里**
3. **复杂任务分步骤**：不要在一次对话里完成所有事
4. **重要指令不要被淹没**：放在最前面
